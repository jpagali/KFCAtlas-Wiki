Recommendation: always-visible search in the mobile navbar

This feels strongest for a knowledge center because it makes the primary job of the site obvious on first paint. The search bar becomes part of the header stack: brand row first, search row second, optional quick links third.

Why it helps Users can search before opening the sidebar, which cuts friction for repeat visitors who already know what they want.
What it changes The navbar becomes a two-row mobile header instead of a single compressed row with hidden utilities.
What to watch We should keep the search field height around 44-48px and make sure result dropdowns do not fight with the off-canvas menu.

Clarity

Best of the two. The first thing users see is the thing they most likely came to do.

Space Cost

Moderate. The header grows taller, especially if we keep quick access chips underneath.

Implementation Risk

Low to medium. Mostly responsive layout and layering work with the existing search component.

Alternative: compact navbar with expandable search

This version is tidier by default, but it slightly weakens discoverability. It’s the safer fallback if we decide the homepage cannot spare the extra vertical room.

Why it helps Preserves a lighter header while still putting search above the main content when activated.
What it changes Search starts as an affordance, then expands inline or overlays the top of the document after tap.
What to watch If the icon-only trigger is too subtle, users may still default to opening the menu instead of searching.

Clarity

Good, but not as strong. Search is available, just not immediately dominant.

Space Cost

Lower. The header stays compact unless the user interacts with search.

Implementation Risk

Medium. State transitions and animation need to feel deliberate to avoid jank.
